# v3.2 Gap Registry

## Purpose

This document is the substrate-development-side accumulator for refinement candidates surfaced during Phase 1 production operation of the substrate. It is **not** substrate-canonical (not a primitive doc; not a spec); it is a working document that v3.2 scope-definition work will consume when the v3.2 release is greenlit.

Per the v2.9.0 → v3.0 retrospective addendum: the substrate's originally-scoped trajectory closed at v3.1.0. v3.2 is the first post-trajectory release whose scope is downstream of operating evidence rather than upstream specification. This registry is that evidence.

## Framing hypothesis (strengthening across surfaced gaps)

**v3.2's coherent thematic frame may be: "substrate behaviors that defend against subject-project-mode operator-side assumptions and environment-side asymmetries."**

Six of the ten gaps below fit this frame directly; two more fit indirectly; two are documentation or asymmetric-sync-discipline issues that share the frame loosely. The frame consolidates the v3.2 release as a coherent themed refinement rather than a multi-theme catchall, conditional on subsequent Phase 1 chains surfacing additional gaps in the same family.

## Gap inventory (10 gaps from Phase 1 task 1.1; first chain + recovery work)

### Gap-1: Stale-daemon lifecycle in subject-project mode

**Class:** substrate
**Discovered:** Phase 1 task 1.1 first dispatch; `fnsr.pid` two days stale; daemon running quietly across days picked up appended tasks immediately
**Fit framing hypothesis:** YES
**v3.2 refinement candidate:** Either a daemon-restart-on-staleness mechanism, or operator-explicit daemon lifecycle commands (`state_admin daemon restart`, `state_admin daemon status`), or both. Aaron's lean: operator-explicit lifecycle commands (preserves substrate-vs-procedure pattern: operator declares, substrate executes). Auto-restart machinery is the wrong shape for the substrate's discipline.

### Gap-2: Applier `source_task` not validated at task-append time

**Class:** substrate
**Discovered:** Phase 1 task 1.1; first applier task (055) blocked with `source_not_in_upstream` because `source_task` wasn't in `depends_on`
**Fit framing hypothesis:** YES
**v3.2 refinement candidate:** Extend append-time validation in `state_admin append-tasks` to check `source_task` is in `depends_on` when both are present. Small change; clear scope; small test surface. Aaron noted this is the same family as gap-7 (append-time validation surfacing constraints that should have been caught earlier than runtime).

### Gap-4: No queue-vs-dispatch buffer in subject-project mode

**v3.2 refinement candidate:** Either a `daemon-pause` / `daemon-resume` mechanism, or per-task `dispatch_when` declarations, or both. Aaron's lean: defer specific shape until v3.2 work surfaces what the actual operator workflow needs.

### Gap-6: Mojibake-repair not auto-included when developer is upstream of applier

**Class:** substrate (with structural caveat from gap-7/gap-8 interaction)

### Gap-7: Chain-shape for large-content tasks isn't constrained by substrate

**Class:** substrate
**Discovered:** Phase 1 task 1.1; first mojibake-fix dispatch (059) failed all 3 attempts with `Claude's response exceeded the 32000 output token maximum` because the operator's framing (three large files in one chain) exceeded what a single LLM dispatch can emit. ### Gap-9: Test-runner subprocess stdout capture degrades Unicode on Windows

**Class:** substrate

### Gap-11: Reconnaissance contract output-shape compliance is **scope-dependent**

**Class:** substrate (LLM-instruction-fidelity boundary; scope-conditional)
**Discovered:** Phase 1 task 1.1 closure Step 2b; first reconnaissance dispatch (079) emitted a single finding's fields (`id`, `claim`, `evidence`, `kind`) at the top level instead of wrapping them in the required envelope (`outputs.findings[]`, `outputs.summary`, `outputs.evidence_paths`). CPS check correctly vetoed with `agent 'reconnaissance' missing required output keys`.

**Refined empirical framing (3 data points across 2 Phase 1 tasks):** Reconnaissance contract output-shape compliance is **scope-dependent**. Substantive-scope recon dispatches (broad investigation; multiple focus areas; multiple files inspected) comply with envelope structure correctly. Narrow-scope closure-recon dispatches (single closure observation; minimal focus areas) reliably produce finding-content at top level rather than envelope-wrapped form. **Two Phase 1 task closures deferred against this gap pending v3.2 contract-visibility refinement targeting narrow-scope dispatch specifically.**

Empirical evidence:
- Task 1.1 substantive recon (052): substantive scope, ~8 focus areas, multiple SPEC sections → **COMPLIANT**
- Task 1.1 closure recon (079): narrow scope, 1 closure observation, 5 focus areas → **NON-COMPLIANT** (finding-shape at top level)
- Task 1.1 closure recon refined (083): narrow scope, refined inputs with envelope examples → **NON-COMPLIANT** (identical failure shape)
- Task 1.2 substantive recon (087): substantive scope, ~11 focus areas, multiple SPEC sections + IMPLEMENTATION_PLAN + DECISIONS + tests → **COMPLIANT**

Pattern: substantive-scope succeeds 2/2; narrow-scope fails 2/2 (across two operator-input-refinement variations).
**Fit framing hypothesis:** YES (subject-project-mode-exposes-LLM-instruction-fidelity-limits; same family as gap-6 developer non-compliance with precise-encoding instructions)
**v3.2 refinement candidate (Aaron's CP5 framing):** Two candidate shapes worth holding:
  1. **Contract-visibility refinement** — reconnaissance and other shaped-output agents' frontmatter / prompt-template includes explicit envelope examples that the dispatch path always injects. Substrate-side enforcement: contract is communicated to the LLM at dispatch time regardless of operator-input specificity.
  2. **CPS-veto-payload enrichment** — when `missing required_outputs` fires, the veto payload includes the expected envelope structure plus what the LLM actually produced, so operator-side recovery is faster (fewer dispatches required to diagnose the shape mismatch).
Either or both; v3.2 design adjudication can decide. The gap's broader pattern (CPS catches non-compliance correctly but each non-compliance requires operator-side recovery) suggests substrate-side prevention is the right direction: **substrate clarity refinements rather than substrate capability expansions** per Aaron's framing.
**Recovery applied this round:** R2 (abandon 079; dispatch 083 with refined inputs containing explicit envelope-structure examples + non-compliance-failure-mode constraint).

**R2 outcome: FALSIFIED.** Task 083 produced the EXACT same wrong shape as 079 (top-level `id`, `claim`, `evidence`, `kind`; no envelope-wrapped findings array). The "operator-side instruction refinement was sufficient" framing is falsified; the **"substrate-side enforcement is required"** framing is confirmed.

Gap-11 is the canonical example of substrate-side LLM-instruction-fidelity enforcement; the deferred Step 2b is the canonical re-dispatch task that v3.2 enables.

**v3.2 design implication:** the contract-visibility refinement may target the specific narrow-scope failure mode rather than reworking the entire contract. Smaller v3.2 scope than originally anticipated; refinement targets the specific compliance-degradation case where reconnaissance scope is small enough that the LLM collapses the envelope into the finding-content shape.

**Class:** substrate-discipline / operator-process refinement

### Gap-13: TypeScript type errors caught at test-runner step, after applier landed

**Class:** substrate (LLM-knowledge-completeness boundary; third instance after gaps 6 and 12)
**Discovered:** Phase 1 task 1.5 first test-runner dispatch (107) failed at build:tests step with 4 TypeScript type errors in src/emit/turtle.ts and src/emit/n-triples.ts. The errors were N3.js library type-narrowing issues (`Quad_Subject` excludes `Literal`; `Quad_Predicate` is `NamedNode` only). The developer's emitted code compiled in its mental model but did not satisfy `@types/n3`'s declared constraints. Architect ratified structural conformance but doesn't run tsc. The applier landed the files (no schema-level issue with the changes[]). Test-runner caught the build failure AFTER the broken code was on disk.
**Fit framing hypothesis:** YES (substrate clarity refinement family with gaps 6, 11, 12; LLM-instruction-fidelity / LLM-knowledge-completeness sub-family)
**Recovery applied this round:** Forward-correction chain (108-112): recon -> dev (targeted type-guard/assertion fixes) -> rat -> apply -> test-runner. Recovery succeeded; 60/60 tests passed after fix.
**v3.2 refinement candidate (two shapes):**
  1. **Substrate-side tsc-no-emit gate** between applier and test-runner. Adds a new system agent (or extends applier) that runs `tsc --noEmit -p tsconfig.test.json` and refuses applier success when type errors exist. Catches the failure BEFORE landing on disk, allowing for retry with refined inputs rather than forward-correction chain.
  2. **Developer agent contract refinement** (same family as gap-11 contract-visibility refinement): instruct developer to mentally type-check against any imported library types before emitting; include precedent examples of type-narrowing patterns. Substrate-side enforcement is harder; this is the LLM-side discipline refinement.

Both shapes target the same gap; both may be needed. v3.2 design adjudicates priority.

**Cost-of-error this round:** ~$3-5 in dispatches for the recovery chain; the type errors were caught (substrate's chain machinery worked correctly) but at a higher cost than a substrate-side preemptive check would impose.

### Gap-12: Developer non-completeness on devDep additions (@types/* companions)

**Class:** substrate (LLM-knowledge-completeness boundary; second instance after gap-6)
**Discovered:** Phase 1 task 1.2 first test-runner dispatch (091) failed at build:tests step with `TS7016: Could not find a declaration file for module 'n3'`. The developer's proposal correctly added `n3` to devDependencies but missed `@types/n3` (the TypeScript types companion package). The test-runner's structured output cited the exact missing package and suggested the canonical fix (`npm i --save-dev @types/n3`).
**Fit framing hypothesis:** YES (same family as gap-6; substrate clarity refinement candidate; LLM-side knowledge gap that the substrate's developer contract should make explicit)
**Recovery applied this round:** G1 (operator-action via canonical npm tooling: `npm i --save-dev @types/n3`). Test-runner re-dispatched as task 092 to re-validate.
**v3.2 refinement candidate:** Developer agent contract gains a structured check or prompt fragment for devDep additions: "When proposing a new devDependency that isn't itself a `@types/*` package, check whether the package bundles its own TypeScript types (via `package.json` field `types` or `typings`, or `index.d.ts` presence). If types aren't bundled, propose the `@types/*` companion alongside the original package." Same shape as gap-11's contract-visibility refinement — making structural compliance criteria visible to the LLM at dispatch time, not just at veto time.

**Pattern observation across gaps 6 + 12:** LLM agents have systematic knowledge gaps that the substrate's contract should explicitly close. v3.2's developer contract refinement may address multiple instances of this pattern rather than gap-12 in isolation.

**Operator-action precedent confirmed:** Standard tooling commands (`npm i --save-dev <pkg>`, `npm install`, `git commit`) are operator infrastructure; agent-proposed mutations are chain machinery. This boundary was exercised here for the second time after gap-5's `npm install` recovery. Worth documenting in PLAYBOOK as an operator-discipline pattern alongside the four substrate properties already documented in §7.5.
